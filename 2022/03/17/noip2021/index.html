报数

题目背景

无

题目描述

报数游戏是一个广为流传的休闲小游戏。参加游戏的每个人要按一定顺序轮流报数，但如果下一个报的数是 777 的倍数，或十进制表示中含有数字 777，就必须跳过这个数，否则就输掉了游戏。

在一个风和日丽的下午，刚刚结束 SPC20nn 比赛的小 r 和小 z 闲得无聊玩起了这个报数游戏。但在只有两个人玩的情况下计算起来还是比较容易的，因此他们玩了很久也没分出胜负。此时小 z 灵光一闪，决定把这个游戏加强：任何一个十进制中含有数字 777​ 的数，它的所有倍数都不能报出来！

形式化地，设 p(x)p(x)p(x)表示 x的十进制表示中是否含有数字 777 ，若含有则 p(x)=1p(x) = 1p(x)=1 ，否则 p(x)=0p(x) = 0p(x)=0。则一个正整数 xxx 不能被报出，当且仅当存在正整数 yyy 和 zzz ，使得 x=yzx = yzx=yz 且 p(y)=1p(y) = 1p(y)=1。

例如，如果小 r 报出了 666 ，由于 777 不能报，所以小 z 下一个需要报 888；如果小 r 报出了 333333，则由于34=17×234=17\times234=17×2，35=7×535=7\times535=7×5 都不能报，小 z 下一个需要报出 363636 ；如果小 r 报出了 696969，由于 70∼7970\sim7970∼79 的数都含有 777，小 z 下一个需要报出 808080 才行。

现在小 r 的上一个数报出了 x，小 z 想快速算出他下一个数要报多少，不过他很快就发现这个游戏可比原版的游戏难算多了，于是他需要你的帮助。当然，如果小 r 报出的 x 本身是不能报出的，你也要快速反应过来小 r 输了才行。

由于小 r 和小 z 玩了很长时间游戏，你也需要回答小 z 的很多个问题。

输入格式

第一行，一个正整数 T 表示小 z 询问的数量。

接下来 T 行，每行一个正整数 x，表示这一次小 r 报出的数。

输出格式

输出共 T 行，每行一个整数，如果小 r 这一次报出的数是不能报出的，输出 -1 ，否则输出小 z 下一次报出的数是多少。

输入输出样例

输入#1
输出#1
输入#2
输出#2

说明/提示

【样例解释#1】

这一组样例的前 3 次询问在题目描述中已有解释。

对于第 4 次询问，由于 300=75×4300=75\times4300=75×4 ，而 757575 中含有 777 ，所以小 r 直接输掉了游戏。

数据范围

对于 10%10\%10% 的数据，T≤10T \leq 10T≤10 ，x≤100x \leq 100x≤100。
对于 30%30\%30% 的数据，T≤100T \leq 100T≤100 ，x≤1000x \leq 1000x≤1000。
对于 50%50\%50% 的数据，T≤1000T \leq 1000T≤1000 ，x≤10000x \leq 10000x≤10000。
对于 70%70\%70% 的数据，T≤10000T \leq 10000T≤10000 ，x≤2×105x \leq 2 \times {10}^5x≤2×105。
对于 100%100\%100% 的数据，1≤T≤2×1051 \le T \leq 2 \times {10}^51≤T≤2×105 ，1≤x≤1071 \le x \leq {10}^71≤x≤107。

分析

挺水一道题，但是也出现了些小问题，一开始想着只要预处理一下把含7的数的倍数都筛选出来即可，然后从 xxx 开始找不是含7的数的倍数，每次询问的复杂度大概都在100以内，总体复杂度大概 O(x+100×T)O(x+100\times T)O(x+100×T) ，测试之后发现预处理直接开最大值都能过，但是最后30%30\%30%的数据超时。

最后稍微修改了一下预处理，使得每次询问的复杂度为 O(1)O(1)O(1) 。

还有个小细节，虽然 x≤107x\le10^7x≤107 ，但是下一个含7的倍数可能超范围。
